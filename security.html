Verification and Disclosure

Security

LNES-01 is an early Mainnet-Beta MVP. Verify every address, use small amounts first, and treat all tooling as active development infrastructure.

EARLY MVP

ExergyNet has executed a confirmed settlement proof path and publishes public verification metadata. It has not yet completed a formal third-party security audit. Do not route funds you cannot afford to lose.

Before you sign

Required verification checklist

  • 01 Verify the Program ID matches Fe8KhdiFWhKcPWH2N2Svqc3VSpK9EzN8nMh9pQ3cPCeD.
  • 02 Verify token mint, vault, and destination accounts before signing any transaction.
  • 03 Verify the transaction instruction, accounts, fees, and expected settlement route.
  • 04 Use small test amounts before routing larger value.
  • 05 Reject any SDK, MCP server, or website copy that changes the Program ID or metadata origin.
Security boundaries

What is protected and what is not yet protected.

Published Program ID

The canonical program reference is public and should be verified against Solscan before use.

Machine-readable metadata

Metadata endpoints are published for agents and indexers, including llms.txt and the well-known ExergyNet manifest.

Explorer telemetry

The L0 explorer displays live producer and block state. It is operational telemetry, not a substitute for cryptographic audit.

Current limitation

Prototype node identities are file-based. Production hardening should use cryptographic keypairs and signed block submissions.

Disclosure

Responsible reporting

Report suspected vulnerabilities with the affected endpoint, transaction, account, reproduction path, and expected impact. Do not exploit live users or attempt unauthorized fund movement.

Security priority is determined by first principles: fund safety, key safety, transaction correctness, replay resistance, metadata integrity, and agent discovery integrity.
